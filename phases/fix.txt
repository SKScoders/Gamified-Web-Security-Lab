1. The actual problem to solve

The dashboard currently shows hardcoded/fake data instead of real per-user progress:

Total Points — hardcoded
Solved Challenges count — hardcoded
Streak — hardcoded
Leaderboard — contains fake test entries (e.g. "PDF Complete", "Final Test", "Phase9 Test", "Security Test") mixed with real users
"Continue where you left off" — shows a stale/impossible state ("Authentication Bypass — 1 attempt(s) in progress") for brand-new accounts that have made zero attempts

Everything below fixes this, end to end, on the real schema.

PHASE 1 — Audit (do this first, produce reports, do not code yet)

Scan the actual repo (frontend + backend + Prisma schema) and produce:

PROJECT_AUDIT.md — broken pages, missing API routes, wrong Prisma queries, hardcoded values, fake/seed data left in production tables, UI inconsistencies, missing loading/error states, unused files/components
SECURITY_REPORT.md — JWT handling, authorization checks (esp. any client-trusted fields — this app has a deliberate one in Level 4's lab, do not "fix" that one, it's the challenge itself), CORS config, rate limiting, input validation, secrets in logs
PERFORMANCE_REPORT.md — dashboard load time, leaderboard query cost, N+1 queries, missing indexes, bundle size

Stop and present these three reports before starting Phase 2. Do not proceed on assumption — wait for confirmation the audit findings are accurate.

PHASE 2 — Dashboard: real data, no mocks
A. Total Points

GET /api/user/points → { totalPoints: number } Computed as SUM(points) across the user's solved challenges via a real Prisma query (aggregate, not fetch-all-then-sum in JS unless the table is tiny).

B. Solved Challenges

GET /api/user/stats → { solved: number, total: number } Dashboard renders solved/total (e.g. 2/4).

C. Streak

Fields on the user's stats record: currentStreak, longestStreak, lastSolvedDate. Logic:

Solving ≥1 challenge on a calendar day the user hadn't already solved one on → increment currentStreak, update longestStreak if exceeded.
If the gap since lastSolvedDate exceeds 24h → reset currentStreak to 1 (not 0 — the qualifying solve that triggered the check counts).
Store dates in UTC; be explicit about timezone handling in the implementation notes since "24 hours" and "calendar day" are different definitions — pick one, document it, apply it consistently.
D. Leaderboard

GET /api/leaderboard, sorted by total points DESC, then completion time ASC. Returns rank, username, score, levels solved, time.

Delete the fake seed entries ("PDF Complete", "Final Test", "Phase9 Test", "Security Test") from the actual database — write a one-time cleanup script, run it, and confirm via a fresh query that they're gone (not just removed from a seed file that already ran).
Real-time refresh: use the existing backend setup — if Socket.IO is already wired in anywhere in this project, extend it; if not, do not introduce a new real-time dependency for this alone — a 15–30s poll is an acceptable, much lower-risk choice given this is a hackathon deliverable on free-tier hosting. Flag the tradeoff and let the user decide before implementing.
E. Continue Where You Left Off

Logic:

attempts.length === 0 → "No active challenges" / "Start your first challenge"
attempts.length > 0 AND most recent attempted challenge's status != solved → show that challenge as "continue"
All challenges solved → "Congratulations! Chain Complete."
Root-cause the current bug (new accounts showing a phantom in-progress attempt) rather than patching around it — this is almost certainly either a seed/fixture record attached to new users by default, or a query missing a WHERE userId = ... filter. Identify which, fix the actual cause, and confirm in the audit report which one it was.
